# work 0020 — [1.0] E05: UnsafePointer non-null + Optional 패턴

- **status**: done

- **cycle**: [1.0] Mojo 1.0.0b1 검증
- **plane**: MOJO-30

## 1) 무엇을 하고자 하는지

release notes:
> "`UnsafePointer` is now non-null by design; use `Optional[UnsafePointer[...]]` for nullable pointers."

→ 0.26 SIMD/메모리 코드의 `xs.unsafe_ptr()` 호출이 1.0에서도 동작하는지, nullable이 필요할 때 `Optional[UnsafePointer[T, _]]` 패턴이 어떻게 보이는지 검증.

## 2) 수행한 일

```mojo
# 01: 0.26 syntax 그대로 (from memory import UnsafePointer)
var p = xs.unsafe_ptr()
var sum: Int = 0
while i < len(xs): sum += p[i]; i += 1

# 03: Optional[UnsafePointer[Int, _]] 시도
def maybe_ptr(yes: Bool) -> Optional[UnsafePointer[Int, _]]:
    if yes: ... else: return None

# 04: std. prefix migration
from std.memory import UnsafePointer
var p = xs.unsafe_ptr()  # 동작?
```

## 3) 예상되는 결과

- 01 → 동작 (단 `from memory` deprecation warning).
- 04 → 정상.
- 03 → Optional[UnsafePointer]은 nullable 표현 가능. 단, origin parameter 처리가 까다로울 수 있음.

## 4) 실제 결과

| 케이스 | 결과 |
|--------|------|
| 01 (`from memory import UnsafePointer`) | ⚠️ warning + 정상 동작. `sum = 150`. |
| 04 (`from std.memory import UnsafePointer`) | ✅ warning 없이 정상. `sum = 150`. |
| 03 (`Optional[UnsafePointer[Int, _]]`) | ❌ `'UnsafePointer[Int, ?]' is not concrete, use '[]' to bind missing parameters` |

### 부수 발견 — `from memory` import도 deprecated

```
01_026_unsafe_ptr.mojo:1:6: warning: Implicit standard library imports are deprecated and will be removed in a future release; fully qualify with 'std.' instead
from memory import UnsafePointer
     ^
     std.
```

→ stdlib 모든 모듈에 `std.` prefix 필요. 0.26 코드의 `from memory`/`from collections`/`from python` 모두 마이그레이션 필요. (0.26 시점에 `from std.python`은 이미 권장됐었음.)

### Optional[UnsafePointer] 시그너처 까다로움

`UnsafePointer`는 1.0에서 origin parameter (lifetime tracking)까지 binding 필요. `_` 와일드카드도 *완전 unbound 표현*과 호환 안 됨 — 추가 명시가 필요한 듯. 본 실험 시간 내 정확한 시그너처 미파악. 후속 work 후보.

## 5) 결론

- **핫루프 unsafe_ptr() 호출**은 1.0에서 *그대로 동작* — SIMD 코드 회귀에 직접 영향 없음. 단 import 문구 수정만 필요 (`from std.memory import`).
- **non-null 변화의 직접 영향은 *흔한 패턴에서는 미미***. 0.26 코드 대부분이 `xs.unsafe_ptr()` 같은 *비-null 보장된 출처*에서 가져옴 → null 가능성을 표현할 일 자체가 없었음.
- **nullable이 정말 필요한 경우** (`malloc 실패 처리` 등): `Optional[UnsafePointer[T, _]]` 시그너처가 까다로워짐. 정확한 binding syntax는 본 cycle에서 미정복 (origin parameter binding이 핵심) — 후속.
- **마이그레이션 비용 (E05 한정)**: import 문구 수정 sed 한 줄. 본질적으로는 0.26과 동일 패턴.

후속:
- `Optional[UnsafePointer[T, ?]]`의 정확한 origin binding syntax
- 0.26 work 0009/0011/0012/0013의 SIMD 코드 — `from memory`/`from std.memory` 일괄 치환 필요 (E06에서 일괄 처리)
