## 5) 결론

### 5.1 ONNX 사용 패턴 (실용)

| 경로 | 가능? | 비고 |
|------|:--:|------|
| Mojo + Python interop + onnxruntime | ✅ | 정상. work 0015 패턴 유지 |
| Mojo + max.engine으로 .onnx 직접 로드 | ❌ | work 0015와 동일 (`session.load(onnx)` 미수용) |
| **Mojo 코드 안에서 `max.engine` import** | **❌ 런타임 crash** | *예상치 못한 발견* — MLIR context 충돌 |
| 별도 Python 스크립트 (Mojo 무관) → onnxruntime | ✅ | 가장 안전 |
| 별도 Python 스크립트 → max.engine + .onnx | ❌ | max.engine은 .onnx 미수용 |

### 5.2 결정적 발견 — Mojo와 MAX의 *공존 한계*

**Mojo 코드에서 max.engine을 Python interop으로 호출하려는 시도는 *crash*** ⚠️.

이는:
1. **Stef의 forum 발언 직접 증거**: "MLIR linking complexities — 동일 MLIR에 link되는 두 라이브러리 동시 사용 불가"
2. **Mojo에서 MAX 활용 유일 방법은 *별도 프로세스***: `max serve`를 띄우고 HTTP 호출하거나, *순수 Python 스크립트*에서 `max.engine` 사용 (Mojo 무관)
3. **본 학습자가 work 0029에서 시도한 `03_max_engine_python_api.py`도 *순수 Python에서 실행해야* 안전**. Mojo 안에서 호출 X.

### 5.3 마이그레이션 영향

- 0.26 work 0014/0015의 `Python.import_module("max.engine")` 패턴은 1.0/26.3에서 *crash까지 진행* (0.26에선 호출은 됐지만 모델 로드에서 실패한 것으로 보임 — 본 cycle에서 더 깊은 layer 차단)
- *실용 결론*: Mojo 안에서 MAX engine 호출 *권장하지 않음*. Python 스크립트 분리.

### 5.4 ONNX 권장 패턴 (CPU 사용자)

```python
# Python 스크립트 — Mojo 무관
import onnxruntime as ort
import numpy as np
sess = ort.InferenceSession("model.onnx")
result = sess.run(None, {"input": np.random.randn(1,3,640,640).astype(np.float32)})
```

Mojo 핫커널이 필요하면:
1. *별도 Mojo 모듈* compile → `.so`로
2. Python에서 `import mojo_so_module`
3. onnxruntime + Mojo so 모듈 *Python에서 합치기*

→ Mojo + MAX는 *언어 두 개를 한 프로세스에 *공존*시키는 사용 패턴은 1.0/26.3에서 *crash***.
