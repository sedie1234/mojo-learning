# work 0032 — [1.0] NumPy 호환성

- **status**: done
- **date**: 2026-05-11
- **plane**: MOJO-42

## 1) 무엇을 하고자 하는지

NumPy ndarray를 1.0에서 (a) 다루기 (b) 변환 비용 (c) attribute 접근 (d) zero-copy 가능성을 확인.

## 2) 수행한 일

`01_numpy.mojo` — ndarray 생성/reshape/operation, list→np.array 변환 비용 (N=10~100K), 속성 접근, item() PyObject 추출.

## 3) 예상되는 결과

0.26 work 0006의 비용 모델 (~16-22 ns/elem)이 1.0에서도 동일. 모든 ndarray API가 PyObject로 동작 (Mojo 1.0이 ndarray *native type 지원* 추가는 미예상).

## 4) 실제 결과

### 4.1 np.array(list) 변환 비용

| N | 0.26 ns/elem | **1.0 ns/elem** | 변화 |
|---|--:|--:|--:|
| 10 | 81 | 80 | 동일 |
| 100 | 26 | 26 | 동일 |
| 1,000 | 22 | 22 | 동일 |
| 10,000 | 21 | 21 | 동일 |
| 100,000 | 16 | 16 | 동일 |

→ **bit-perfect 동일**. 마샬링 비용 모델 *완전 보존*.

### 4.2 ndarray 동작 ✅

- `np.arange(10)`, `np.ones((3,3))` 정상
- `arr.shape`, `arr.dtype`, `arr.size`, `arr.ndim` 모두 PyObject 반환
- `arr.item((1,2))` → PyObject → `Int(py=)` 변환 정상
- `arr.tolist()` 양방향 list 변환 정상
- `x + y`, `np.dot(x, y)` vector op 정상

### 4.3 numpy 버전

1.0 venv에는 **numpy 2.4.4** (0.26 venv에는 2.0.x). NumPy 자체의 메이저 변화가 있지만 *interop 표면*은 영향 없음.

### 4.4 Zero-copy 가능성 — 미정복

- ndarray는 *PyObject 그대로* 다뤄짐. *Mojo native 타입으로의 zero-copy view*는 본 cycle에서 미정복.
- 0.26 work 0006의 TODO "NumPy ndarray ↔ Mojo Tensor zero-copy via buffer protocol"이 1.0에서도 *미정복 상태*.
- 추정 진입점: `from std.memory import UnsafePointer` + numpy `__array_interface__["data"]`로 raw pointer 받기. 후속 work 후보.

## 5) 결론

- **NumPy interop 비용 모델 1.0 ↔ 0.26 *bit-perfect 동일***.
- **API 표면 변화 없음**: 모든 ndarray 작업이 PyObject로 동일하게 동작.
- **Zero-copy 진입점은 1.0에서도 *미정복*** — TODO 유지.
- **마이그레이션 비용 = 0**. 0.26 NumPy 사용 코드 *그대로 동작*.
