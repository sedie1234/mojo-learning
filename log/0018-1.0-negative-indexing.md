# work 0018 — [1.0] E03: Negative indexing 제거 검증

- **status**: done
- **date**: 2026-05-08
- **cycle**: [1.0] Mojo 1.0.0b1 검증
- **plane**: MOJO-28
- **code**: `experiments/0018-1.0-negative-indexing/`

## 1) 무엇을 하고자 하는지

release notes 항목 "negative indexing removed from all collections — `x[-1]` now compile-time error"의 직접 검증. 추가로 마이그레이션 패턴 `x[len(x)-1]`이 정상 동작하는지.

## 2) 수행한 일

```mojo
# 04_list_literal.mojo — List 리터럴 + 음수 index
var xs: List[Int] = [10, 20, 30, 40, 50]
var last_neg = xs[-1]                  # ← 1.0에서 에러 기대

# 05_list_literal_fixed.mojo — 마이그레이션 패턴
var last = xs[len(xs) - 1]             # ← 권장 형태

# 06_string_byte_keyword.mojo
var c = s[byte=0]                      # 새 String API 추정
```

## 3) 예상되는 결과

- `xs[-1]` → compile error.
- `xs[len(xs)-1]` → 정상.
- String도 동일.

## 4) 실제 결과

### 4.1 부수적 발견 — List 가변 인자 생성자 *완전 제거*

```mojo
var xs = List[Int](10, 20, 30, 40, 50)   # ← 0.26 syntax
```

→ 1.0에서 컴파일 에러. note 제시된 *유효 cosntructor 후보*:
- `List[T](capacity=N)`
- `List[T](length=N, fill=...)`
- `List[T](__list_literal__=...)` ← `[a,b,c]` 리터럴이 내부적으로 호출
- `List[T](unsafe_uninit_length=N)`
- `List[T](copy=...)`
- `List[T](take=...)` ← move 인입
- 모두 **keyword-only**

→ **`var xs: List[Int] = [a,b,c]`** 가 1.0의 정공법. **literal syntax가 first-class**가 됨.

### 4.2 negative index — 정확한 에러 메시지

```
04_list_literal.mojo:5:22: note: call expansion failed
oss/modular/mojo/stdlib/std/collections/list.mojo:1303:9:
  note: constraint failed: negative indexing is not supported,
        use e.g. `x[len(x) - 1]`
```

→ ✅ 컴파일 에러. **에러 메시지에 마이그레이션 힌트 포함** — 매우 친절.

### 4.3 마이그레이션 패턴 동작

```
=== 05: ===
len = 5
last = 50
first = 10
```

→ ✅ `xs[len(xs) - 1]` 정상.

### 4.4 String — `[byte=N]` keyword 인덱싱

```
=== 06: s[byte=0] ===
h
```

→ ✅ 동작. release notes의 *grapheme cluster support*와 일관 — String 인덱싱이 *byte vs grapheme* 명시 keyword로 분리됨. 0.26의 `s[i]` 무명 인덱싱은 1.0에서 keyword 강제.

## 5) 결론

본 실험에서 *부수적으로 더 큰 발견*이 나옴:

| 변화 | 영향 |
|------|------|
| **Negative indexing 제거** (예상) | 명확한 에러 + 힌트. 마이그레이션 *기계적*. |
| **List 가변 인자 생성자 제거** (부수) | `List[T](a,b,c)` → `List[T] = [a,b,c]` 리터럴 또는 keyword-only ctor. **모든 예제 코드 영향**. |
| **String 인덱싱 keyword 강제** (부수) | `s[i]` → `s[byte=i]` / `s[grapheme=i]`. 의미적으로 더 정확하지만 코드 변경 비용. |

**Migration cost 평가** (E03 한정 + 부수 발견 포함):
- negative index → `len(x)-1`: sed/manual로 10분 작업
- `List[T](a,b,c)` → `[a,b,c]`: regex 가능, 30분~1시간 (literal context 차이로 일부 수동)
- `s[i]` → `s[byte=i]` 또는 `s[grapheme=i]`: **각 사용처 *의미* 판단 필요** — 자동화 어렵고 *byte vs grapheme*을 의도적으로 결정해야 함

→ String 변경이 가장 까다로움. *byte 의도였는지 grapheme 의도였는지*는 코드 작성자만 안다.

후속:
- `s[grapheme=N]` 정확 동작 검증 (한글 등 multi-byte char에서)
- List 새 ctor의 capacity/length+fill/copy/take 패턴 어느 게 *권장* 패턴인지
