## 1) 무엇을 하고자 하는지

Mojo 1.0 GPU programming (**SIMT** 모델) 의 *대표 패턴 3개*를 코드로 보여주는 것:
1. **Vector add** — 1D kernel, 가장 단순한 SIMT 패턴.
2. **Sum reduction** — shared memory tree reduction + warp.sum primitive.
3. **Tiled matmul** — 2D shared memory blocking + barrier 동기화.

사용자 요청: *실제 실행은 안 하고 코드만 작성 + GPU 사용법 소개*. modular/skills `mojo-gpu-fundamentals` skill의 정확한 패턴 따름.

## 2) 수행한 일

세 파일 작성:
- `01_gpu_vector_add.mojo` — 1D kernel
- `02_gpu_reduction.mojo` — block tree reduction + warp shuffle
- `03_gpu_matmul.mojo` — 2D tiled matmul with shared memory

핵심 1.0 GPU 패턴 (CUDA와의 차이):

| CUDA | Mojo 1.0 |
|------|---------|
| `__global__ void kernel(...)` | plain `def kernel(...)` — 데코레이터 *없음* |
| `kernel<<<grid, block>>>(args)` | `ctx.enqueue_function[kernel](args, grid_dim=..., block_dim=...)` |
| `cudaMalloc(&p, n*sizeof(T))` | `ctx.enqueue_create_buffer[dtype](n)` |
| `cudaMemcpy(...)` | `ctx.enqueue_copy(dst_buf, src_buf)` (또는 keyword) |
| `cudaDeviceSynchronize()` | `ctx.synchronize()` (또는 `map_to_host()` context manager 자동) |
| `__syncthreads()` | `barrier()` from `std.gpu.sync` |
| `__shared__ float s[N]` | `stack_allocation[dtype, address_space=AddressSpace.SHARED](row_major[...]())` |
| `threadIdx.x`, `blockIdx.x*blockDim.x+threadIdx.x` | `thread_idx.x`, `global_idx.x` |
| `__shfl_down_sync(...)` / `warp reduction` | `warp.shuffle_down(...)`, `warp.sum(...)`, `warp.reduce(...)` |
| `atomicAdd(&ptr, val)` | `Atomic.fetch_add(ptr, val)` from `std.atomic` |
| raw `float*` kernel args | `TileTensor[dtype, LayoutType, MutAnyOrigin]` |
| `cudaFree(ptr)` | 자동 (scope 종료 시) |

## 3) 예상되는 결과

- CPU only 머신 (본 환경): `has_accelerator()=false`이므로 *GPU 호출 부분이 comptime에서 제외*되어 정상 컴파일 + "No GPU" 메시지만 출력될 것.
- GPU 머신: 정상 컴파일 + 실행 결과 산출.

## 4) 실제 결과

**CPU only 머신에서 컴파일 자체 불가** — *예상 외 발견*:

```
01_gpu_vector_add.mojo: error: no matching method in call to 'enqueue_function'
note: candidate not viable: 'enqueue_function' parameter 'Ts' has 'DevicePassable' type
note: candidate not viable: failed to infer parameter 'signature_func'
```

원인:
- `comptime if has_accelerator(): ... else: ... `로 가드 두어도, 컴파일러는 *type-check를 항상 수행*.
- `enqueue_function[kernel]`이 kernel을 *DevicePassable* trait conform하는 type으로 받는데, CPU only 머신에선 `DeviceContext`/`enqueue_function` 자체가 *no-op 또는 비활성 type*이 되어 *signature_func* 추론이 실패.
- `comptime if` 가드는 *실행 분기*는 제거하지만 *parsing + type-check*는 *모든 분기*에 적용됨.

→ **CPU only 머신에선 *GPU 코드를 한 파일 안에 둘 수 없음***. 옵션:
1. GPU 머신에서 컴파일 (정공법 — 사용자도 이걸 인지하고 "실행 안 함" 명시)
2. GPU 코드를 *별도 모듈*로 분리하고 *조건부 import* (build-time guard) — 본 cycle 미시도
3. `#if GPU` 같은 *전처리기 macro* (Mojo 1.0에 없음)

## 5) 결론

### 5.1 코드 자체는 정확한 1.0 SIMT 패턴

3개 코드 파일은 modular/skills `mojo-gpu-fundamentals` SKILL의 공식 권장 패턴을 *그대로* 따름. GPU 머신에서 실행하면 정상 작동할 것.

### 5.2 1.0 GPU 모델의 학습 포인트 (코드에서 확인 가능)

1. **No CUDA syntax**: `__global__`, `<<<>>>`, `__shared__` 모두 *Mojo 키워드 X*. 일반 `def`로 kernel, `ctx.enqueue_function[kernel]`로 발사.
2. **TileTensor as primary GPU data abstraction**: raw `float*` 대신 *layout-aware tensor* (rank/dim 컴파일 시점 추적).
3. **comptime layout system**: `row_major[N]()` 또는 `row_major[M, K]()` — 차원이 컴파일 시점 상수.
4. **layout assert 필수**: `comptime assert tensor.flat_rank == N`을 *kernel 진입 시 항상* 적어야 N-D indexing 활성.
5. **Shared memory via `stack_allocation`**: `address_space=AddressSpace.SHARED` 인자로 shared scope 명시.
6. **Warp primitives**: `warp.sum`, `warp.max`, `warp.shuffle_down` — hardware shuffle 활용.
7. **Atomic.fetch_add**: 마지막 block 단계 결과를 global output에 누적.
8. **map_to_host() context manager**: 자동 synchronize + host-pointer 노출. `cudaMemcpy + cudaDeviceSynchronize`의 한 줄 합성.

### 5.3 본 cycle 한계 명시

- CPU only 머신: GPU 코드 컴파일 불가 → *문서 가치만*
- 정확성 검증: GPU 머신 도착 후 first action
- 실측 성능 (matmul GFLOPS, BW 등): 후속 cycle

### 5.4 작성된 GPU 패턴 활용성

- 본 코드들은 GPU 도착 시 *그대로* 컴파일/실행 가능 (modular/skills 공식 패턴이므로)
- 1.0 GPU programming의 *학습 진입점*으로 가치 있음 — C++/CUDA 사용자에게 매핑 표가 가장 빠른 학습 경로

후속 (GPU 시):
- 컴파일/실행 검증 + ASM dump (PTX/SASS) — CPU의 `vaddps %zmm` 검증의 GPU 대응
- vector add: CPU SIMD16 1.0의 160 GB/s vs GPU H100 1-2 TB/s
- matmul: CPU 1.0 blocked 268 GFLOPS @ N=1024 vs GPU
- reduction: tree reduction vs warp.sum hardware shuffle 효율
