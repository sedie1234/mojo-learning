# work 0024 — [1.0] E09: Safe closure ref capture + parallelize

- **status**: partial (회귀는 OK, 명시 capture syntax 미정복)
- **date**: 2026-05-08
- **cycle**: [1.0] Mojo 1.0.0b1 검증
- **plane**: MOJO-34

## 1) 무엇을 하고자 하는지

release notes:
> "Unified closures with `ref` capture convention support"
> "Stateless closures auto-lift to top-level functions and work as FFI callbacks"
> `unified` keyword removed; specify unified-closure semantics with explicit capture lists `{...}`"

→ 0.26 work 0012의 `@parameter fn task` + `parallelize[task](n)` 패턴이 1.0에서 동작하는지, 새 명시 capture list syntax가 어떻게 보이는지 검증.

## 2) 수행한 일

```mojo
# 01: 0.26 패턴 그대로 (@parameter fn)
@parameter
fn task(i: Int):
    var s = 0
    for j in range(1, 1001): s += i * j
    p[i] = s
parallelize[task](n)

# 02: fn → def 마이그레이션
@parameter
def task(i: Int):
    ...
parallelize[task](n)

# 03: 명시 capture {p} 시도
@parameter
def task[capture: {p}](i: Int):
    ...
```

알고리즘: i ∈ [0,1000), j ∈ [1,1000]에 대해 sum(i*j). 정답 = (∑j) × (∑i) = 500500 × 499500 = 249,999,750,000.

## 3) 예상되는 결과

- 01 (0.26 fn) → fn warning + 정상 동작.
- 02 (def) → 정상 동작.
- 03 (명시 capture) → release notes의 "capture lists {...}" 가이드 따라 동작.

## 4) 실제 결과

| 시도 | 결과 |
|------|------|
| 01 (0.26 `@parameter fn`) | ✅ fn warning + `total = 249999750000` (정답) |
| 02 (`@parameter def`) | ✅ warning 없이 `total = 249999750000` |
| 03 (`def task[capture: {p}]`) | ❌ `{p}` 가 *Set literal*로 parsed → "Set 생성자 candidate not viable" 에러 |

### 핵심 회귀 결과 — 좋음

0.26 패턴 (`@parameter fn task` + `parallelize[task](n)`)이 1.0에서 *그대로* 동작. fn warning만 발생, 정답 일치.

### 명시 capture syntax — 미정복

`def task[capture: {p}](i: Int)` 시도 시 `{p}`가 *Set literal* (`{element_1, element_2}` 형태)로 파싱돼 컴파일 에러:

```
note: 'KeyElement' is aka 'Copyable & Hashable & Equatable'
    def task[capture: {p}](i: Int):
                       ^
```

→ 1.0의 *capture list 명시 syntax 정확 형식*은 release notes 표현으로는 추정 불가. 본 cycle 미정복. `unified` 키워드도 폐지됐고, 사용자 측 진입점(`@unified`, `@capture(...)`, `def f[caps={...}]`, `def f({...}) -> ...` 등) 어떤 데코레이터/syntax인지 미파악.

## 5) 결론

**핵심 결과는 양호**:
- **0.26의 `@parameter fn`/`@parameter def` + `parallelize[task](n)` 패턴이 1.0에서 그대로 작동**
- 즉 *0.26 work 0012/0013 (matmul + parallelize 가속)의 핵심 코드 마이그레이션 비용은 *키워드 fn→def만***
- safe closure는 *기본 구현*에서 충분히 안전 (parallelize 호출이 정상 종료, 결과 정답)

**미해결 부분**:
- 명시 capture list `{...}` syntax — release notes 가이드로는 *Set literal과 충돌*해 사용자 측 정확 진입점 미파악
- `ref` capture convention의 사용자 코드 진입점
- "stateless closures auto-lift to top-level"이 *암묵적*으로 작동하는지 (extra syntax 없이도)

→ 가장 중요한 *0.26 코드 회귀는 완전*. 새 기능의 syntax 표면은 *beta 시기 docs 부족*으로 불완전.
