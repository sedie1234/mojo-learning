### 4.1 Naive / SIMD16 / parallelize (work 0012 회귀)

| N | 1.0 naive GFLOPS | 1.0 SIMD16 GFLOPS | **1.0 SIMD+par GFLOPS** | 0.26 SIMD+par (work 0012) |
|---|--:|--:|--:|--:|
| 128 | 4.25 | 50.6 | **175.3** | 315 (work 0012 결과 — 측정 변동 가능) |
| 256 | 3.34 | 65.7 | **354.6** | — |
| 512 | (skipped) | 83.5 | **440.3** | **437** |
| 1024 | (skipped) | 78.0 | **357.9** | — |

→ N=512 par GFLOPS = **0.26 437 ↔ 1.0 440** *동등*. 1.0 단순 회귀 OK.

→ N=128에서 1.0이 0.26 대비 작게 측정된 이유는 *0.26 work 0012*에서 N=128이 가장 좋은 GFLOPS를 기록한 측정값(315 GFLOPS)이 thermal/load 영향으로 변동된 케이스일 가능성. work 0012의 N=512 437이 reproducible reference.

### 4.2 Blocked matmul (work 0013 회귀) — **1.0이 모든 N에서 빠름**

| N | 0.26 blocked (work 0013) | **1.0 blocked** | 변화 |
|---|--:|--:|--:|
| 512 | 171 GFLOPS | **186** | +9% |
| 1024 | 210 GFLOPS | **268** | **+28%** |
| 2048 | 236 GFLOPS | **257** | +9% |

→ **N=1024에서 1.0 +28%**. 다른 N도 일관적으로 +9% 이상. SIMD vector add E06에서 본 *1.0 컴파일러 backend 개선*과 같은 패턴.

→ 단 NumPy MKL과의 격차는 여전 (work 0013 NumPy MKL N=2048 = 1540 GFLOPS). MKL 6× 격차는 *register tiling micro-kernel* 부재 때문 (TODO).

## 5) 결론

- **Matmul 회귀 OK**: SIMD16+par는 0.26 동등, blocked는 1.0이 +9~28% 개선.
- **컴파일러 backend 개선 일관 패턴**: vector add(0021) → reduction(0011) → matmul(0026) 모두 1.0이 *동등 ~ 약간 빠름*. LLVM/MLIR 업데이트 효과로 추정.
- **parallelize import 경로 변경 없음**: vector add는 `sort`가 prelude builtin으로 이동했지만, `parallelize`는 `from std.algorithm.functional`에 그대로. *모든 stdlib symbol이 일괄 이동한 게 아님*.
- **N=1024 blocked +28%가 가장 흥미**: L3 capacity 경계 영역에서 *cache-aware loop transform이 더 잘 작동*. LLVM의 LoopVectorize 또는 MemorySSA 개선 추정.
- **MKL 격차는 1.0에서도 그대로**: 6× 차이는 *알고리즘 영역* (register tiling 부재) — 컴파일러 backend 개선만으로는 못 메움. TODO 유지.
