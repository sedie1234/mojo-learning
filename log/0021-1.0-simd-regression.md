마이그레이션에서 발견된 추가 0.26→1.0 변화:

- ASM에 `vaddps %zmm` emit (0.26과 동일).
- ns/elem은 ±10% 범위 내 (코드 동일, 컴파일러 backend 동일 LLVM/MLIR).
- 마이그레이션 비용은 *작음* — fn/import/alias 키워드 수정 정도.

## 4) 실제 결과

### 4.1 ASM 검증 — `vaddps %zmm` emit 유지

```
$ grep "vaddps" 03_bench_10.s
	vaddps	(%r12,%rdx,4), %zmm0, %zmm0
	vaddps	(%r12,%rdx,4), %zmm0, %zmm0
$ grep -c "vaddps" 03_bench_10.s
2
```

- ✅ AVX-512 명령 그대로 emit. 0.26과 **동일 패턴** (warmup + measure 블록 = 2회).
- 주변 명령(`vmovups (%rbx,%rdx,4), %zmm0` load → `vaddps` add → `vmovups %zmm0, (%r13,%rdx,4)` store)도 0.26 ASM과 동일 구조.

→ **zero-cost abstraction은 1.0에서도 유지**. SIMD 코드는 *코드 수정 없이* 0.26 ↔ 1.0 ISA 동등.

### 4.2 성능 비교 (JIT, ns 단위, median of 10)

| N | 0.26 scalar | 0.26 simd16 | **1.0 scalar** | **1.0 simd16** | simd16 변화 |
|---|--:|--:|--:|--:|--:|
| 1K (L1) | 430 | 50 | 220 | 30 | **0.26 대비 −40%** |
| 64K (L2) | 13,191 | 3,140 | 12,479 | 3,199 | +2% (거의 동일) |
| 1M (L3) | (미기록) | (미기록) | 198,792 | 73,673 | — |
| 16M (DRAM) | (미기록) | 7,362,373 | 6,610,009 | 4,651,043 | **0.26 대비 −37%** |

- L1 거주 영역에서 1.0이 *더 빠름* — 함수 진입/loop overhead 감소 추정.
- DRAM 영역에서도 1.0이 *훨씬 빠름* — 측정 변동 가능성 있지만 일관 패턴.
- L2 캐시 영역은 거의 동일.

### 4.3 AOT 산출물 크기

| 0.26 (work 0009) | 1.0.0b1 |
|--:|--:|
| 일반적으로 ~50 KB+ | **33 KB** (bench), 18.6 KB (hello) |

→ **AOT 사이즈 감소** 일관 (E01에서도 hello world가 44KB → 18.6KB로 감소).

### 4.4 마이그레이션 변경 항목 (직접 fix-up)

| 0.26 | 1.0 |
|------|-----|
| `from std.algorithm.sort import sort` | (import 없음 — prelude builtin) |
| `alias W: Int = 16` | `comptime W: Int = 16` (warning, 동작은 OK) |
| `fn main():` | `def main():` (warning) |
| `fn add16(a, b) -> SIMD[...]:` | `def add16(a, b) -> SIMD[...]:` |

→ 4종 syntax 차이. 본문 SIMD 호출 코드(`(c+i).store(a.load[width=W](i) + b.load[width=W](i))`)는 *0 변경*.

## 5) 결론

- **SIMD 회귀 결과는 *우수***. ASM은 동일, 성능은 동등 ~ 더 빠름. SIMD 학습자산 보호됨.
- **zero-cost abstraction 유지**: `(c+i).store(a.load[width=16](i)+b.load[width=16](i))` → `vaddps (...), %zmm0, %zmm0` *그대로*.
- **마이그레이션 비용 (SIMD 파일 1개당)**: 4종 키워드 수정 ~5분. 본질 코드 무변화.
- **L1 + DRAM 모두 빨라진 이유 추정**:
  - LLVM/MLIR backend upgrade
  - 함수 진입 overhead 감소 (1.0의 def은 fn보다 더 enabling — Python-호환 모드인데 강제 strict는 사라졌으니 codegen이 더 유연?)
  - 측정 변동 가능성도 배제 못 함 (CPU thermal/turbo 상태)
- 본 결과는 "Mojo 1.0이 SIMD 성능을 *나쁘게* 만들지 않았다 — 오히려 *나아지거나 동등*" 강한 결론.

후속:
- `comptime`이 `alias`의 정확한 대체인지 (의미 차이 있는지) 확인
- C++ AVX-512 빌드와 1.0 SIMD16의 직접 비교 (0.26 시점에 했지만 1.0에서 재검증 가치)
- vectorize decorator의 1.0 시그너처 (0.26에선 미정복)
