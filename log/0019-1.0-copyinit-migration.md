## 1) 무엇을 하고자 하는지

release notes:
> "Legacy `__moveinit__()` and `__copyinit__()` auto-rewriting removed; rename to `__init__()` with keyword-only arguments."

→ 0.26 시점에 작성한 `__copyinit__(out self, copy: Self)` 패턴이 1.0에서 *실제로* 어떻게 깨지는지, 새 keyword-only `__init__` 패턴이 어떻게 보이는지, 컴파일러 자동 생성이 어디까지 가능한지를 직접 검증.

## 2) 수행한 일

```mojo
# 01_026_pattern.mojo — 0.26 그대로
struct Box(Copyable):
    var x: Int
    fn __init__(out self, x: Int): self.x = x
    fn __copyinit__(out self, copy: Self):     # 0.26 syntax
        self.x = copy.x

# 02_10_keyword_only_init.mojo — 1.0 새 패턴
struct Box(Copyable):
    var x: Int
    fn __init__(out self, x: Int): self.x = x
    fn __init__(out self, *, copy: Self):      # keyword-only
        self.x = copy.x

# 03_minimal.mojo — copy ctor 생략, Copyable conform만
struct Box(Copyable):
    var x: Int
    fn __init__(out self, x: Int): self.x = x
    # copy ctor 정의 없음
```

## 3) 예상되는 결과

- 01 → 컴파일 에러 (auto-rewriting 제거 명시).
- 02 → 정상 동작.
- 03 → 단순 Pod-like 타입은 컴파일러 자동 생성될 가능성 (Rust `#[derive(Copy)]`처럼).

## 4) 실제 결과

| 케이스 | 결과 |
|--------|------|
| 01 (0.26 `__copyinit__`) | ❌ `error: 'None' has no attributes — self.x = copy.x` |
| 02 (1.0 keyword-only `__init__`) | ✅ `7` 출력 |
| 03 (ctor 정의 없음, Copyable만) | ✅ `7` 출력 — **컴파일러 자동 copy ctor 생성** |

`fn` deprecation warning은 모든 케이스에 표시.

### 01의 정확한 에러

```
01_026_pattern.mojo:7:13: error: 'None' has no attributes
        self.x = copy.x
        ~~~~^
```

→ `__copyinit__`가 더 이상 *특별한 dunder* 가 아니라 *일반 method 시그니처*로 처리됨 → `out self`만 사용한다는 뜻으로 해석되어 `copy` 파라미터가 None 취급. 즉 **0.26 코드는 *조용히 깨지지 않고 컴파일 에러***로 명확히 거부.

### 03의 의미

```mojo
struct Box(Copyable):
    var x: Int
    fn __init__(out self, x: Int): self.x = x
```

→ `Copyable` conform 선언만 하면 *컴파일러가 자동으로 copy ctor 생성*. 이건 0.26보다 **훨씬 깔끔**. C++의 default copy ctor 자동 생성과 의미 일치.

## 5) 결론

- **0.26 `__copyinit__/__moveinit__` 패턴은 1.0에서 *완전히 깨짐*** — auto-rewriting 폐지가 명백.
- **마이그레이션 비용 (0.26 코드 기준)**: 모든 struct의 dunder를 다시 봐야 함.
  - 단순 case (Pod-like) → 03 패턴: dunder 두 개 *삭제*만으로 자동 생성 활용 가능.
  - 깊은 copy 등 사용자 정의 필요 → 02 패턴: dunder를 `__init__(out self, *, copy: Self)` 또는 `(out self, *, take: Self)`로 *rename + keyword-only* 변환.
- **에러 메시지 친절도**: copy ctor 케이스의 에러는 *원인을 직접 가리키지 않음* (`'None' has no attributes`만 나옴) — 0.26 코드를 처음 받은 사람은 헤맬 것. release notes를 안 읽으면 디버깅 시간 길 수 있음.
- **장점**: keyword-only `*, copy:`/`*, take:`로 통일된 시그니처는 *전체 ctor 정책이 한눈에 보임*. C++의 *5+ rule of zero/three/five*보다 명료.

**Migration cost (1 struct당)**:
- Pod-like: -2 줄 (dunder 두 개 제거) — *오히려 코드 줄어듦*
- Deep-copy 필요: rename + keyword-only `*` 추가 = ~3분/struct

**가장 큰 함정**: dunder rename을 안 한 채 `*, copy:` 키워드만 추가하면 *조용히 새 overload만 추가*되고 기존 dunder는 무력화 → 동작은 정상이지만 dunder 두 개가 dead code로 남음. **수동 grep + 마이그레이션 권장**.

## 6) 추가 검증 (사용자 질문 — 2026-05-11): struct 내부 메서드도 def으로 가야 하나?

본 실험 01-03 파일에서 struct 내부 메서드는 *`fn __init__` / `fn __copyinit__`*으로 작성했음. 이는 0.26 패턴 demo 목적이었으나, 그대로 두면 *struct 내부의 fn에도 동일하게 deprecation warning*이 출력됨:

```
01_026_pattern.mojo:4:5: warning: 'fn' is deprecated, use 'def' instead
    fn __init__(out self, x: Int):
    ^~
    def
```

→ **struct 메서드라고 fn이 살아남는 special case 없음**. 모든 위치에서 def 권장.

추가 실험 `04_full_def.mojo` / `05_minimal_def.mojo` 작성:

```mojo
struct Box(Copyable):
    var x: Int
    def __init__(out self, x: Int): self.x = x
    def __init__(out self, *, copy: Self): self.x = copy.x

def main():
    var a = Box(7); var b = a.copy()
    print(a.x, b.x)   # → 7 7  (warning 없음)
```

→ ✅ **warning 0, 결과 동일**. struct 내부 메서드도 def으로 일괄 변환이 1.0 권장 형태.

**결론 보완**:
- *모든 fn → def* (top-level + struct method 모두). work 0017 E02의 `sed 's/^fn /def /g'`는 *struct 내부는 미커버* (들여쓰기 안 fn은 `^fn` 매칭 안 됨). 권장 sed: `sed -E 's/(^|[[:space:]])fn /\1def /g'` 또는 `sed 's/\bfn /def /g'`.

→ E02 마이그레이션 sed 명령을 *그대로* 적용하면 struct 내부 fn 누락. **다음 cycle에서 일괄 마이그레이션 시 word-boundary 사용 필수**.
