- **status**: partial (사용자 측 syntax 미정복)

- **plane**: MOJO-33

## 1) 무엇을 하고자 하는지

release notes:
> "Conditional conformance to traits."
> "Standard library types now use conditional conformances."
> "`conforms_to()` expressions drive refinement, reducing need for `trait_downcast`"

→ Rust의 `impl<T: Bound> Trait for X<T>`에 해당하는 Mojo 1.0의 *conditional conformance*가 사용자 코드에서 어떻게 보이는지 최소 예제로 검증.

## 2) 수행한 일

여러 syntax 시도:

```mojo
# 시도 01: 메서드에 where 절 (release notes 예시)
struct Box[T: Copyable]:
    def describe(self) -> String where T: Stringable:
        return "Box(" + self.x.to_str() + ")"

# 시도 02: impl ... where ... (release notes 예시)
impl Box[T] as Stringable where T: Stringable:
    def to_str(self) -> String: ...

# 시도 03: comptime if T.conforms_to[Trait]()
def describe(self) -> String:
    comptime if T.conforms_to[Stringable]():
        return ...

# 시도 04: stdlib의 conditional 동작 검증 (Tuple)
var t: Tuple[Int, Int] = Tuple[Int, Int]()  # Defaultable
```

## 3) 예상되는 결과

WebFetch 추정 syntax (`where T: Stringable`)가 동작.

## 4) 실제 결과

| 시도 | 결과 |
|------|------|
| 01 (`where T: Stringable` on method) | ❌ "Copyable does not implement '__bool__' method" — `where`가 *boolean expression*으로 해석됨, type bound 표현 아님 |
| 02 (`impl Box[T] as Trait where ...`) | ❌ "expressions are not allowed at global scope" — `impl` 키워드는 global scope에서 안 통함 |
| 03 (`comptime if T.conforms_to[...]()`) | ❌ `T` 자리에 `Self.T` 써야 + Copyable instance에 conforms_to 메서드 없음 |
| 04 (stdlib Tuple conditional 동작) | ✅ `Tuple[Int, Int]()` → `0 0` — *stdlib* 안에서는 conditional conformance 동작 |

### 핵심 발견 — 사용자 측 syntax는 release notes 표현과 다름

release notes의 예시 코드:
```mojo
impl Tuple[*Ts] as Defaultable where *Ts: Defaultable:
    fn __init__(out self): ...
```

이건 *문서용 의사 syntax* 또는 *stdlib 내부 syntax*일 가능성. **사용자 코드에서 직접 작성하는 정확한 syntax는 본 cycle에서 미정복**:
- `where T: Stringable`은 *type bound이 아닌 boolean expression* (Mojo 1.0의 `where`는 *comptime predicate*)
- `T.conforms_to[Trait]()`도 instance method가 아니라 *type-level metafunction* 형태가 필요해 보이나 정확 syntax 미파악
- method-level `where`도 boolean predicate 방식이라 trait bound로 직접 표현 불가
- `impl ... where ...` 패턴은 외부 docs에서 인용되지만 실제 컴파일러는 거부

stdlib 내부에서는 *작동 중*이 확인됨 (Tuple Defaultable conditional). 그러나 그 문법이 사용자에게 *어떤 형태로 노출되는지*가 본 cycle에서 미해결.

## 5) 결론

- **stdlib에 conditional conformance가 *적용되어 있음*은 검증** — Tuple이 모든 element가 Defaultable일 때 default 생성 가능.
- **사용자 코드에서 직접 conditional conformance를 *선언*하는 정확 syntax는 본 cycle 미정복**. release notes의 예시 syntax(`where T: Bound`, `impl X as T where ...`)가 컴파일러에 *그대로* 받아들여지지 않음.
- **추정 원인**:
  - 1.0의 `where`는 *comptime boolean expression*이지 type bound 표현이 아님 (release notes의 "Compiler narrows types from `where` clauses and `comptime if/assert`"와 일관)
  - conditional conformance의 사용자 측 진입점은 *별도 키워드*(가능성: `requires`, `@conditional`, `comptime if`)일 수 있음
- **본 결과의 정직한 평가**: *기능은 1.0에 들어와 있으나, 사용자 측 syntax 표면이 release notes로는 정확 추정 불가*. 공식 docs/예제가 아직 충분치 않은 듯 — beta 시기 한계.
