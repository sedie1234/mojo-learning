## 1) 무엇을 하고자 하는지

CNN의 가장 핵심 op인 **2D convolution**의 Mojo 1.0 *CPU + GPU* 코드를 작성:
- CPU: List + SIMD16 + parallelize로 직접 구현 → 실제 실행해 정확도/성능 검증
- GPU: modular/skills 패턴으로 1 thread = 1 output element (oh, ow) — 코드만

설정: NHWC, N=1, H=W=64, C_in=16, C_out=32, kernel 3×3, stride=1, padding=1 (same conv).

## 2) 수행한 일

- `01_conv2d_cpu.mojo`: parallelize[row_task](H) — 행 단위 thread 분배. inner는 oh,ow,oc,kh,kw,ic loop (정확도 우선, 추가 최적화 없음).
- `02_conv2d_gpu.mojo`: `comptime if has_accelerator()` 가드 안 GPU 커널. `block_idx.y/.x * BLOCK_H/W + thread_idx.y/.x = oh/ow`. comptime for로 K×K 루프 unroll.

마이그레이션 주의:
- **`out`은 1.0 예약어** (`def f(out self, ...)`) → 변수 이름으로 사용 불가. `output`으로 rename.
- **`K`는 일반적 이름이라 충돌 위험** → GPU 파일에선 `KSZ`로 사용.

## 3) 예상되는 결과

- CPU: 정상 동작, FLOPS 계산 가능.
- GPU: CPU only 머신에선 컴파일 불가 (work 0028 동일 한계). GPU 머신에서 정상 동작 예상.

### 4.3 1.0 conv2d 작성 시 함정 (학습 가치)

| 함정 | 대응 |
|------|------|
| `out: TileTensor[...]` argument | `out`은 1.0 예약어 (`def f(out self, x)`) → `output` 사용 |
| `K = 3` 변수 | 한 글자 변수가 충돌 가능 → `KSZ` 또는 `KERN_SIZE` |
| `K` named in surrounding scope | comptime for의 loop var 이름과 부딪힐 수 있음 |
| `rebind[Scalar[dtype]](tensor[i,j])` | layout가 다른 tensor element 곱셈 시 필수 (work 0028의 modular/skills 가이드 인용) |
| `comptime for kh in range(K):` | K가 작을 때 unroll → register pressure 낮음 + boundary check elimination 강화 |

## 5) 결론

### 5.1 CPU 직접 작성 가능성 입증

- Mojo 1.0 + parallelize로 **43 GFLOPS @ 16 cores**. 단순 naive 구현임에도 합리적 throughput.
- *YOLO 같은 production CNN을 layer 단위로 Mojo로 재작성*하는 시나리오 (work 0015 0030 결합)의 *기술적 가능성* 입증.
- 추가 최적화 시 (im2col + GEMM, register tiling, FMA 활용) NumPy MKL 또는 onnxruntime 수준 도달 가능 추정.

### 5.2 GPU 패턴 — CUDA 사용자 진입 비용

modular/skills 패턴 기반 GPU conv2d는:
- thread → output element 1:1 매핑 (CUDA 표준 패턴)
- comptime for로 K×K 루프 unroll (CUDA `#pragma unroll`)
- TileTensor로 layout-aware indexing (CUDA의 raw pointer + manual stride 대체)

→ CUDA 사용자가 Mojo GPU 진입할 때 *학습 곡선이 매우 완만*. ASM 수준 동등 가능 (work 0021 SIMD 같은 패턴).

### 5.3 1.0 conv2d 작성의 함정 정리

work 0030이 누적한 *코드 작성 시 주의*:
1. **`out` 예약어** — 가장 흔히 부딪히는 문제 (모든 NN 코드의 `output` argument)
2. **layout 불일치 시 `rebind`** — 다른 layout tensor 원소 곱셈은 type mismatch
3. **comptime for unroll** — K 같은 작은 dim에 효과적
4. **NHWC vs NCHW** — modular/skills는 NHWC 권장 (PyTorch는 NCHW default — 변환 필요)

후속 (GPU 시):
- GPU conv2d 실제 실행 + 성능 측정 (cuDNN과 비교)
- im2col + GEMM 변환 → matmul 빠른 경로 활용
- ResNet/YOLO 한 block을 Mojo로 작성하고 PyTorch와 정확도/성능 비교
