## 1) 무엇을 하고자 하는지

release notes:
> "Bounds checking enabled by default on CPU."

→ List subscript (`xs[i]`)에 자동으로 bounds check가 들어가게 됨. 핫 루프에서 *얼마의 overhead*인지, *unsafe_ptr() 직접 사용 대비 차이*가 얼마인지 직접 측정. 끄는 옵션 있는지도 살펴봄.

## 2) 수행한 일

같은 sum reduction을 두 패턴으로 측정:
- (a) `xs[i]` (List subscript, 1.0에선 bounds-checked)
- (b) `p[i]` where `p = xs.unsafe_ptr()` (raw pointer, no check)

```mojo
for i in range(n): s += xs[i]   # (a)
for i in range(n): s += p[i]    # (b)
```

워밍업 3 + median of 10. 각 N=1K, 64K, 1M, 16M.

추가: ASM dump로 bounds-check 실패 경로(panic/unreachable/abort) 출현 확인.

## 3) 예상되는 결과

- subscript는 unsafe_ptr 대비 *몇 % 느릴 것*. release notes의 "default on"은 안전성을 비용으로 사는 것이니까.
- 또는 LLVM의 *bounds check hoisting*이 잘 작동하면 거의 동일 (단, 컴파일러가 i가 [0,n) 범위임을 증명해야).

## 4) 실제 결과

| N | subscript ns | unsafe ns | overhead |
|---|--:|--:|:--:|
| 1,024 | 210 | 210 | **0%** |
| 65,536 | 11,899 | 11,889 | **0%** |
| 1,048,576 | 190,595 | 190,065 | **0%** |
| 16,777,216 | 3,379,909 | 3,331,232 | **1%** |

→ **Overhead 0~1%**. 측정 노이즈 수준.

ASM 분석:
- 100 KB AOT ASM 안에 *bounds-check 실패 경로 (panic/unreachable/abort/trap) **0회 등장***
- 즉 LLVM이 모든 subscript의 bounds check를 *완전히 hoist out / 제거*

## 5) 결론

**Bounds check default-on은 *공짜에 가까움***. LLVM이 *iteration variable의 range를 증명*하면 check를 제거.

본 패턴 (`for i in range(n): xs[i]`)은 *증명 가능한 가장 단순한 케이스*:
- `i ∈ [0, n)` 정적 보장
- `len(xs) >= n` 추적 가능 (xs가 length=n으로 alloc됨)
- LLVM의 `LoopVectorize` + `IndVarSimplify`가 check를 모두 elimination

**한계 — 본 결과의 일반화**:
- *동적/외부 데이터 인덱스*: 예 `xs[external_input_value]` → check가 남을 가능성
- *비-선형 인덱스 패턴*: 예 `xs[hash(...) % n]` → 증명 어려움
- *SIMD load[width=16]에서의 boundary*: SIMD 영역에선 unsafe_ptr 사용이 정공법 (bounds check가 SIMD로 fold 안 될 가능성)

→ 본 머신/패턴 한정해서는 *List subscript와 unsafe_ptr이 동등 성능*. **0.26의 unsafe_ptr 핫루프 패턴(work 0009)을 1.0에선 *List subscript로 회귀해도 무방***. 단 SIMD load의 width 경계는 여전히 unsafe_ptr 사용.

**개발자 경험 변화**:
- 0.26: "성능 위해 unsafe_ptr 써야 함" + "범위 실수하면 silent UB"
- 1.0: "List subscript도 같은 성능" + "범위 실수하면 *runtime check가 잡아줌*"
- 즉 *안전성 + 성능을 한 번에 가짐*. 큰 진보.

후속:
- 동적 인덱스 패턴에서 bounds check 명시적 cost 측정
- `--no-bounds-check` 같은 컴파일 플래그 존재 여부 (본 cycle에서 미파악)
- SIMD `load[width=W]`의 경계 처리 변화
