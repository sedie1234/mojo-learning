## 3) 예상되는 결과

- **parallelize 효과**: matmul (work 0026)에서 5-6× speedup 관찰. vector add는 *ops/byte ≈ 1/12*로 메모리 바운드 → parallelize 효과 *적거나 음수*일 것 (roofline 이론).
- **명시 capture**: SKILL.md 예시 `def task(i: Int) {read p}:`로 작동.

## 4) 실제 결과

### 4.1 SIMD16 + parallelize 효과

| N | 영역 | single SIMD16 | par16 SIMD16 | speedup | bw(par) |
|---|------|--:|--:|:--:|--:|
| 1,048,576 | L3 거주 | 78 μs (160 GB/s) | **237 μs (53 GB/s)** | **0× (역효과)** | — |
| 16,777,216 | DRAM | 4.58 ms (43 GB/s) | 4.30 ms (46 GB/s) | 1.07× | — |
| 67,108,864 | DRAM | 19.7 ms (40 GB/s) | 19.0 ms (42 GB/s) | 1.04× | — |

→ **parallelize가 *역효과* 또는 *효과 미미***.

### 4.2 명시 capture syntax 재시도

| 시도 | 결과 |
|------|:--:|
| `@parameter def task(i: Int) {read p, read n}:` (nested) | ❌ `expected ':' in function definition` |
| top-level `def task(i: Int) {read p}:` + `parallelize[task](n)` | ❌ |
| top-level `def task(i: Int):` 자동 capture + `parallelize[task](n)` | ❌ `missing num_workers/num_work_items` |

원인 파악:
- `parallelize[task](n)`은 task를 **comptime parameter**로 받음 → task가 `@parameter` 데코레이터 가져야 함
- modular/skills의 `def closure(i: Int) {read p}:` + `vectorize[simd_width](size, closure)` 예시는 **vectorize**가 closure를 *runtime arg*로 받는 형태 — parallelize와 호출 컨벤션이 다름
- 즉 명시 capture `{...}` syntax는 *runtime-arg API*용 (vectorize 등)이고, *parallelize는 여전히 `@parameter def` 패턴* 사용 — auto-capture

→ **명시 capture syntax 자체는 1.0에서 정확히 작동**하지만, parallelize에는 적용 불가. work 0024의 미정복은 *호출 컨벤션의 미스매치*였고, 명시 capture *기능* 자체는 살아있음.

## 5) 결론

### 5.1 ops/byte 차이가 parallelize 효과를 결정 — *직접 증명*

| 알고리즘 | ops/byte | parallelize 효과 |
|---------|--:|:--:|
| matmul (work 0026, N=512, 캐시 거주) | ~85 | **5-6×** |
| vector add (본 work, DRAM) | 0.083 | **거의 없음 (1×)** |
| vector add (본 work, L3 거주) | 0.083 | **역효과 (0.3×)** |

→ work 0013의 cache_hierarchy 이론 (*roofline의 BW-bound 영역에선 코어 추가 무의미*) **직접 증명**. *학습 자산으로 매우 강력*.

### 5.2 parallelize 역효과 원인 (N=1M)

- 작업 본문은 1M element vector add → ~78 μs
- 16 chunk로 쪼개면 chunk당 ~5 μs
- parallelize task dispatch overhead ~10-15 μs/chunk × 16 chunks = ~160-240 μs (관측치 237 μs와 일치)
- → **dispatch overhead > 작업 시간** → parallelize *역효과*

**일반 원칙**: parallelize는 *코어당 작업이 *충분히 큰* 경우*에만 효과. 작은 작업 + 메모리 바운드 패턴엔 *single-thread + SIMD가 정답*.

### 5.3 명시 capture syntax 1.0 사용 가이드 (확정)

| API | 호출 컨벤션 | capture 방식 |
|-----|------------|--------------|
| `parallelize[task](n)` | task = comptime param | `@parameter def task` + auto-capture |
| `vectorize[W](n, closure)` | closure = runtime arg | top-level `def closure(i: Int) {read p, mut x}:` 명시 capture |
| `Bench.iter_custom` | callable runtime | top-level `def` + `{...}` 명시 capture |

→ work 0024의 ⚠️ → ✅ **명시 capture syntax는 *vectorize/iter_custom 같은 runtime-arg API*에 적용**. parallelize는 0.26부터 1.0까지 `@parameter` 패턴 유지.

### 5.4 회귀 평가

- single-thread SIMD16 N=1M: 0.26 `73 μs` ↔ 1.0 `78 μs` → **동등** (측정 변동)
- single-thread SIMD16 N=16M: 0.26 `7.36 ms` ↔ 1.0 `4.58 ms` → **0.26 대비 -38%** (work 0021과 일관, *L3 → DRAM 영역에서 1.0 backend 개선*)
