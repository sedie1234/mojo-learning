# work 0017 — [1.0] E02: fn → def 마이그레이션

- **status**: done
- **date**: 2026-05-08
- **cycle**: [1.0] Mojo 1.0.0b1 검증
- **plane**: MOJO-27
- **code**: `experiments/0017-1.0-fn-to-def/`

## 1) 무엇을 하고자 하는지

`fn` 키워드가 deprecated 됐다. 0.26 시점에 strict mode (`fn`)으로 작성한 함수들을 `def`로 마이그레이션 시 (a) 동일 동작, (b) 동일 성능, (c) sed 한 줄 치환 가능 여부 확인.

## 2) 수행한 일

세 파일:
- `01_fn_style.mojo` — `fn add` / `fn sum_to` / `fn main` (모든 인자/반환 타입 명시)
- `02_def_style.mojo` — 동일 로직, 모든 `fn` → `def`
- `03_sed_migrated.mojo` — `sed 's/^fn /def /g' 01_fn_style.mojo`

```bash
mojo run 01_fn_style.mojo   # warning + 정상 실행
mojo run 02_def_style.mojo  # warning 없이 정상 실행
mojo run 03_sed_migrated.mojo
diff 02_def_style.mojo 03_sed_migrated.mojo
```

## 3) 예상되는 결과

- fn 코드는 def으로 *기계적*으로 치환 가능할 것 (의미가 동일하다면).
- diff는 주석/공백 차이만 남을 것.
- 동일 결과(`add(3,4)=7`, `sum_to(100)=5050`).

## 4) 실제 결과

```
01_fn_style.mojo:2:1: warning: 'fn' is deprecated, use 'def' instead
fn add(a: Int, b: Int) -> Int:
^~
def
01_fn_style.mojo:5:1: warning: 'fn' is deprecated, use 'def' instead
01_fn_style.mojo:13:1: warning: 'fn' is deprecated, use 'def' instead
add(3,4) = 7
sum_to(100) = 5050
```

- ✅ 모든 `fn` 출현 위치에 warning + fix-it 출력 (커서까지 표시).
- ✅ 동일 결과 3종 동일.
- ✅ `sed 's/^fn /def /g'`만으로 100% 마이그레이션 — diff는 의도적으로 다른 주석 한 줄뿐.

## 5) 결론

- **단순 함수 정의에 한해서는 fn → def 마이그레이션이 *완전히 기계적***. sed 한 줄로 안전.
- 단, **0.26의 fn은 *strict mode* (모든 타입 명시 권장), def은 *Python-호환 mode* (생략 가능)**라는 의미 차이가 컴파일러 안에서는 사라짐 — 1.0에선 둘 다 같은 codegen 경로로 보임 (warning 외 차이 없음).
- 즉 **1.0의 def은 0.26의 fn + def을 합친 것**. strict-vs-permissive 구분 자체가 사라진 듯. 의미론적으로는 *def은 더 풍부해진 것*.
- 큰 코드베이스도 sed/find로 안전 마이그레이션 가능. **migration cost ≈ 0**.

후속:
- struct method (`fn __init__`, `fn method`) 영역에서도 마이그레이션이 깨끗한지는 E04에서 검증.
- 반환 타입 미명시 def이 PythonObject로 falling back 하는 경우의 정확한 시그너처는 후속.
